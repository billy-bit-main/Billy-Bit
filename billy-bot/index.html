Billy Bot is basically a mini Alexa — a local voice assistant that uses keyword extraction instead of AI. It runs on a Raspberry Pi with Python 3. There are two core output functions: play(), which plays a pre-recorded voice clip of me saying an answer (because I'm a narcissist), and say(), which falls back to text-to-speech when the response needs dynamic data.

Billy waits for the wake word "billy" before listening for commands. Then it runs speech-to-text through Google (with Vosk as an offline fallback) and converts what you said into a string. From there it scans for keywords. On the surface it looks like AI — but it's actually just a very long if/elif chain checking whether certain words exist in the sentence.

Before you click Write, click the cog icon (Edit Settings). This is what makes the headless setup actually work:

  • Set hostname → e.g. billybot (you will use this later)
  • Enable SSH → use password authentication (do not skip this)
  • Set username and password → this becomes [username] everywhere in this guide
  • Configure wireless LAN → exact WiFi name + password (case sensitive). Country code: AU
  • Set locale → your timezone

If you miss any of these, the Pi will boot but you won’t be able to connect to it.

If you can't find it in your router, run this from your computer's terminal:

ping billybot.local

If that fails, wait another 30–60 seconds and try again. First boot can be slow and the network comes online last.

Open a terminal on your computer:

ssh [username]@billybot.local

This usually works without needing the IP. If it doesn't, use the IP address instead.

If nothing happens, check for errors immediately — do not keep retrying blindly:

python3 assistant.py

The delays (sleep) are not random — they prevent race conditions during boot (WiFi + audio not ready yet). Removing them will break startup reliability.

ERR_001
PyAudio on Python 3.14
Completely incompatible. Would not build, no useful errors. Fixed by avoiding PyAudio directly and relying on system-level portaudio + SpeechRecognition instead.
ERR_002
pipwin broken
Tried using pipwin to install PyAudio on Windows. Didn't work. Abandoned immediately and moved everything onto the Pi instead.
ERR_003
playsound abandoned
Unreliable, blocking, and inconsistent across platforms. Switched to pygame for full control over playback.
ERR_004
mp4 not supported by pygame
pygame does not support mp4 audio. Converted everything to wav. Problem solved instantly.
ERR_005
espeak not installed
pyttsx3 ran without errors but produced no sound. Root cause: missing espeak packages.
ERR_006
FLAC missing
SpeechRecognition failed silently. Installing flac fixed everything instantly.
ERR_007
Bluetooth audio chaos
Pairing worked, audio didn’t. Default sink wrong. PulseAudio timing issues. Took hours. Fixed with manual sink assignment and boot delays.
ERR_008
Vosk AcceptWaveform typo
Single capitalisation mistake. Completely broke recognition. Took too long to notice.
ERR_009
Case sensitivity
Worked on Windows, failed on Linux. Filename mismatch. Fixed by matching exact casing.

I started this knowing basically nothing about Linux, audio systems, or speech recognition. The first time anything worked — like actually hearing a response come back — felt unreal. Not because it was complex, but because it proved the entire chain worked end to end.

The biggest shift was going from randomly trying things to actually debugging properly. Early problems took days. Later ones took an hour. Same difficulty — just better thinking.

From the outside, this looks like a voice assistant. It isn't. It's a structured system of rules, conditions, and audio playback pretending to be something smarter. But it works. It's sitting there. And that gap between “this is simple” and “this actually works” is the whole project.
